# server 开发规范

适用范围：`server/` 目录及其子文件。

## 当前项目结构

- 当前是 Go 后端项目，入口在 `cmd/main.go`。
- 使用 `gin` 提供 HTTP 服务。
- 配置文件为 `config.yaml`。
- 主要目录职责：
  - `cmd/`：启动入口与入口测试。
  - `internal/router/`：路由与鉴权中间件。
  - `internal/ctrl/`：请求接收、参数处理、响应组装。
  - `internal/service/`：业务逻辑。
  - `internal/model/`：请求体、响应体、文件数据库模型。
  - `internal/config/`：配置读取。
  - `internal/logger/`：日志能力。
  - `pkg/`：第三方能力封装，目前包含 `scribble` 文本数据库管理。

## 改动原则

- 新功能优先沿用 `router -> ctrl -> service -> model` 的分层，不跨层直接串逻辑。
- HTTP 返回优先复用 `internal/model/response`，不要在各处手写不一致的响应格式。
- 日志统一走 `internal/logger`，不要直接散落 `fmt.Println` 作为正式日志。
- 配置项变更时，同步检查 `config.yaml`、配置读取逻辑和使用位置是否一致。
- 涉及路由、JWT、请求参数或响应结构的调整，必须同时检查调用链和测试影响。
- 文件数据库相关改动要关注 `db.filedir`、读写路径和本地数据兼容性，避免破坏现有数据。

## 常用命令

以下命令基于仓库现状确认可用，执行目录为 `server/`：

```bash
go run ./cmd --conf ./config.yaml
go test ./...
```

## 验证要求

- 后端改动后，至少执行一次 `go test ./...`。
- 如果改动影响启动流程、配置读取、路由注册或中间件，还要补跑 `go run ./cmd --conf ./config.yaml` 做启动验证。
- 如果改动接口行为，至少补一项针对性验证：现有测试、补充测试，或明确记录手工验证结果。
- 测试或启动失败时，先定位根因，再决定是修代码、修配置还是修测试；不要直接跳过。

## 代码要求

- 优先补在现有文件和现有结构里，非必要不要新增新的基础设施层。
- 新增测试优先贴近被测代码；当前仓库已有入口测试，新增逻辑应逐步补充对应测试。
- 处理错误时返回清晰错误信息，避免吞错或只记录不返回。
