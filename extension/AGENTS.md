# extension 开发规范

适用范围：`extension/` 目录及其子文件。

## 当前技术栈

- 当前是原生浏览器插件项目，基于 Chrome Extension Manifest V3。
- 代码直接由浏览器加载，没有 `package.json`、没有构建脚本、没有前端测试脚本。
- 主要文件职责：
  - `manifest.json`：插件声明、权限、入口配置。
  - `background.js`：后台 service worker，负责监听页面与上报数据。
  - `contentScript.js`：页面内容采集。
  - `popup.html` + `popup.js`：插件弹窗界面与交互。
  - `detail.html`：详情页面。
  - `trail.css`：页面样式。
  - `jq.js`：本地引入的 jQuery 运行时。

## 改动原则

- 保持“原生 HTML + CSS + JavaScript + jQuery”的现状，不擅自引入打包器、框架或包管理器。
- 修改插件权限、注入脚本、后台监听逻辑时，先确认 `manifest.json`、`background.js`、`contentScript.js` 三者是否需要一起调整。
- 前端存储键、登录态字段、接口路径一旦变更，必须同步检查弹窗读取逻辑和后台上报逻辑。
- 不在插件代码里新增与当前任务无关的 UI 框架、工具函数库或全局状态。
- 直接编辑 `jq.js` 风险高，除非是明确升级或修复依赖问题，否则不要改动。

## 编码约定

- 延续现有代码风格，优先局部修改，不做整仓风格漂移。
- DOM 事件、存储读写、消息通信要写清楚触发条件和失败处理。
- 采集逻辑要注意浏览器受限页面，不能假设所有页面都可注入或可读取。
- 权限遵循最小必要原则；新增权限时，需要说明用途和影响范围。

## 验证方式

当前仓库没有前端 `lint`、`test`、`build` 脚本，前端改动以手工验证为主。

- 打开 `chrome://extensions` 或兼容浏览器的扩展管理页，开启开发者模式。
- 通过“加载已解压的扩展程序”加载 `extension/` 目录；改动后执行重新加载。
- 至少验证本次改动直接影响的入口：
  - 改 `popup.*`：确认弹窗能正常打开，核心按钮和表单无明显报错。
  - 改 `background.js` / `contentScript.js`：确认普通 `http/https` 页面可触发采集流程，后台无明显报错。
  - 改 `manifest.json`：确认扩展能成功加载，权限或入口没有导致加载失败。
  - 改样式或详情页：确认页面能正常展示，不出现明显错位或空白。
- 提交结果时，注明本次实际完成了哪些手工验证。
